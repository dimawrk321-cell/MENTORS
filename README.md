# MENTORS

Закрытая менторская платформа подготовки к карьере в ML / DS / NLP / AI Engineering.
Единственный источник требований — [MENTORS_TZ.md](./MENTORS_TZ.md).

## Стек

Next.js 15 (App Router, RSC, Server Actions) · TypeScript strict · Tailwind CSS 4 + дизайн-токены · Radix UI · PostgreSQL 16 · Prisma · Vitest · pino · Nodemailer (email) · node-cron (worker фоновых задач) · Docker Compose + Caddy (прод).

## Требования

- Node.js >= 22.13 (пиновый `pnpm@11.10` требует Node ≥ 22.13; поле `engines.node` в `package.json`)
- pnpm 11.10 (закреплён через `packageManager`; corepack подтянет нужную версию)
- PostgreSQL 16 — либо портативный в папке проекта (без Docker и прав администратора, см. ниже), либо Docker. Кодировка UTF-8, `LC_CTYPE` с корректной кириллицей (не `C`) — иначе trgm-fallback поиска молча пуст (см. changelog ТЗ, раздел 7.11); `scripts/db.mjs` создаёт кластер уже правильно.

## Запуск с нуля (без Docker — основной способ на этой машине)

```bash
pnpm install          # зависимости
cp .env.example .env  # заполнить переменные: как минимум SEED_* (реквизиты owner и двух менторов)
pnpm db:setup         # разово: скачает EDB-бинарники PostgreSQL 16 (~326 МБ) в ./pgsql,
                      # выполнит initdb и создаст базу mentors (пользователь mentors/mentors)
pnpm db:start         # запустить базу (localhost:5432)
pnpm db:migrate       # применить миграции Prisma
pnpm db:seed          # dev-seed: owner + 2 ментора из SEED_* env (идемпотентно)
pnpm dev              # http://localhost:3000 (перед стартом проверяет доступность базы)
```

Вход после сида — реквизиты из `SEED_OWNER_EMAIL` / `SEED_OWNER_PASSWORD` (owner попадает в /admin).
Учеников приглашает админ со страницы «Ученики». Без заданного `SMTP_HOST` почта
работает в режиме `jsonTransport` (письма пишутся в лог, приложение не падает) —
инвайт-ссылка всегда доступна в интерфейсе с кнопкой «Копировать».

Папка `./pgsql` (бинарники, data-каталог, лог) в git не попадает (`.gitignore`).
Остановка — `pnpm db:stop`, проверка — `pnpm db:status`. Если база не запущена,
`pnpm dev` откажется стартовать с подсказкой «База не запущена — выполни pnpm db:start».

прод-VPS, чтобы команда тестировала платформу и вычитывала импортированный контент.
Полная прод-ревизия — этап 13 (см. changelog ТЗ).

- **URL:** https://dev.155-212-211-251.sslip.io (временное имя через sslip.io; TLS — Caddy автоматически). Старое имя `dev.62-113-108-135.sslip.io` временно отдаёт 301-редирект на новое (переезд с IP под ТСПУ-фильтрацией — см. changelog ТЗ).
- **Сервер:** Ubuntu 24.04, `/opt/mentors` (клон репозитория), Docker Compose: `postgres` + `web` + `caddy` + `worker`. `worker` — живой процесс node-cron (все джобы 7.15: слоты, стрик, дайджест, напоминания, YouTube-монитор, флаги, рассылка отложенных писем), поднят этапом 9. Здоровье контейнера — по heartbeat-файлу: воркер трогает его каждые 30с, Docker healthcheck (`worker/healthcheck.mjs`) читает свежесть, так что зависший цикл переводит контейнер в `unhealthy` (этап 12.2).
- **Админ-доступ:** только по SSH через Tailscale (`ssh mentors-vps`). Публичный вход по паролю в sshd отключён; наружу открыты только 80/443.
- **Секреты:** `/opt/mentors/.env.prod` (в git не попадает; шаблон — `.env.prod.example`). Посмотреть на сервере: `cat /opt/mentors/.env.prod`.

Импорт Notion (раздел 7.14 ТЗ) имеет **два пути с общей кодовой базой** —
`buildImportPlan` + `commitPlan` через единый `runImport`; второй логики импорта
нет. Всё создаётся черновиками; `dry-run` — точное зеркало `--commit` по всем
счётчикам отчёта (курсы/модули/уроки/категории/вопросы/**привязки**/гайды).

**1. CLI (`pnpm run import`)** — для крупной первичной загрузки и повторных
прогонов из терминала (полный контроль, отчёт в `import-report.md`, картинки
резолвятся из папки рядом с `.md`). Выполняется локально по SSH-туннелю к
серверной БД (порт postgres наружу закрыт):

```bash
ssh -N -L 5433:127.0.0.1:5432 mentors-vps &   # туннель к postgres контейнера через хост
# затем локально с DATABASE_URL=postgresql://mentors:***@127.0.0.1:5433/mentors :
# (именно `pnpm run import` — `pnpm import` без run вызывает встроенную команду pnpm)
pnpm run import -- --file="import/notion/<export>.md" --dry-run
pnpm run import -- --file="import/notion/<export>.md" --commit
```

**2. Админка (`/admin/import`, admin+)** — для команды без доступа к серверу и
SSH: загрузка `.md`-экспорта (до 25 МБ) + опционально `.zip` с картинками,
кнопки «Dry-run»/«Импортировать», прогресс по фазам (разбор → план → запись,
поллинг статуса), отчёт на экране в структуре CLI-отчёта + «Скачать .md», история
запусков (`import_runs`). Загруженный файл валидируется, кладётся во временный
каталог вне git и удаляется после прогона; каждый запуск — в аудите
(`import.executed`). Параллельный запуск двух импортов запрещён (advisory-lock
`notion-import`); импорт идёт в том же процессе (см. `DECISION` в
`lib/services/notion-import/admin-import.ts`).

Когда какой: массовая заливка/сверка «боем» и точный контроль окружения — CLI;
разовая догрузка, вычитка отчёта и повтор силами менторов через браузер —
админка.

## Эксплуатация (runbook)

Все команды ниже — на сервере (`ssh mentors-vps`), из `/opt/mentors`, где
`C="docker compose --env-file .env.prod -f docker-compose.prod.yml"`. Написано
для владельца: каждая процедура самодостаточна.

### Диагностика и рестарт сервисов

```bash
cd /opt/mentors
C="docker compose --env-file .env.prod -f docker-compose.prod.yml"
$C ps                      # статусы: web/worker/postgres должны быть healthy
$C logs --tail 100 web     # 500-ки, ошибки рендера, миграции (entrypoint)
$C logs --tail 100 worker  # "worker ready" + тики "job done" (11 джоб)
$C logs --tail 100 caddy   # TLS/ACME (выпуск сертификата, редиректы)
$C logs --tail 50 postgres # редко нужно: рестарты, нехватка места
df -h /                    # место: бэкапы + docker-образа копятся
```

Значения health: `web` — HTTP `/api/health` изнутри контейнера; `worker` —
свежесть heartbeat-файла (зависший цикл → `unhealthy`); `postgres` — `pg_isready`.
`caddy` без healthcheck — судить по логам и открыванию сайта.

Рестарты (от мягкого к жёсткому):

```bash
$C restart web         # перезапуск процесса (конфиг/env НЕ перечитываются)
$C up -d web           # пересоздать контейнер (перечитывает .env.prod)
$C up -d --build web   # пересобрать образ и пересоздать (после правок кода)
$C down && $C up -d    # полный перезапуск всего стека (последнее средство;
                       # postgres переживает down — данные в volume mentors_pgdata)
```

Типовые диагнозы: web `unhealthy` сразу после деплоя — смотри миграции в
`logs web` (entrypoint гоняет `prisma migrate deploy`); worker `Up`, но джобы
не тикают — `logs worker | grep -E "job (done|failed)"`; сайт не открывается
при живых контейнерах — `logs caddy` (истёк/не выпустился сертификат) и
проверка доступности по операторам (см. чеклист ниже).

### Бэкапы

Ежедневный `pg_dump | gzip` в `/opt/mentors/backups` — хост-cron
`/etc/cron.d/mentors-backup` (03:30 UTC), скрипт `scripts/backup.sh`, ротация:
последние **14** файлов `mentors-*.sql.gz`, лог — `/var/log/mentors-backup.log`
(строки `backup ok`). Ручные дампы `pre-<событие>-<дата>.dump` ротация не
трогает. Проверка, что бэкапы живы:

```bash
ls -lht /opt/mentors/backups | head -5      # свежий файл датирован сегодняшним 03:30
tail -3 /var/log/mentors-backup.log         # последние "backup ok"
```

Перед любой рискованной операцией (down с удалением volume, массовые правки
данных, restore) — свежий ручной дамп:

```bash
$C exec -T postgres pg_dump -U mentors -d mentors -Fc \
  > backups/pre-<что-делаем>-$(date +%F).dump
```

### Восстановление из бэкапа (проверено боем в заходе 13.2)

Шаг 1 — **всегда** сначала проверочный restore в throwaway-базу (не трогает бой):

```bash
cd /opt/mentors
C="docker compose --env-file .env.prod -f docker-compose.prod.yml"
DUMP=backups/mentors-YYYYMMDD-0330.sql.gz     # выбрать нужный
$C exec -T postgres psql -U mentors -d postgres -c "CREATE DATABASE mentors_restore_check"
gunzip -c "$DUMP" | $C exec -T postgres psql -U mentors -d mentors_restore_check -q -v ON_ERROR_STOP=1
# сверка счётчиков (яд: courses/lessons/questions/guides/users должны совпасть с ожиданием):
for t in users courses lessons questions guides sessions xp_events; do
  echo "$t: $($C exec -T postgres psql -U mentors -d mentors_restore_check -Atc "select count(*) from $t")"
done
```

Шаг 2 — боевой restore (**замещает текущие данные** — сначала ручной дамп боя!):

```bash
$C exec -T postgres pg_dump -U mentors -d mentors -Fc > backups/pre-restore-$(date +%F).dump
$C stop web worker                             # отпустить коннекты к базе
$C exec -T postgres psql -U mentors -d postgres \
  -c "DROP DATABASE mentors WITH (FORCE)" -c "CREATE DATABASE mentors OWNER mentors"
gunzip -c "$DUMP" | $C exec -T postgres psql -U mentors -d mentors -q -v ON_ERROR_STOP=1
$C up -d web worker                            # entrypoint догонит миграции, если дамп старее кода
curl -fsS https://dev.155-212-211-251.sslip.io/api/health   # {"ok":true}
$C exec -T postgres psql -U mentors -d postgres -c "DROP DATABASE IF EXISTS mentors_restore_check"
```

Прогон захода 13.2 (2026-07-22): restore дампа 824K — 0.9 с; 53 таблицы,
справочные счётчики сошлись 1-в-1 (questions 359, guides 22, categories 58,
achievements 19), расхождения только по дневному дрейфу активности и
задокументированным удалениям владельца (сверено с audit_log).

### Ротация ссылок библиотеки

Записи собеседований (раздел 7.9 ТЗ) лежат на Я.Диске, ссылки нужно обновлять
примерно раз в месяц. Механика:

- Пульт (`/admin`) показывает флаг «Записи со ссылкой старше 30 дней: N» —
  считается по `recordings.link_updated_at`.
- Джоба воркера `linkRotationReminder` (1-го числа месяца) шлёт админам
  уведомление о записях с устаревшими ссылками.
- Обновление: `/admin/library` → карточка записи → новая ссылка Я.Диска
  (`url` / `embed_url`). Сохранение проставляет `link_updated_at = now`, и запись
  уходит из флага. Публикация возможна только при полном чеклисте (лица/голос/
  имена/согласие) — гейт встроен в форму.

### Смена IP (отработано на ТСПУ-миграции 2026-07)

Когда часть операторов РФ фильтрует текущий IP (симптом: у одних всё работает,
у других таймаут при живом сервере):

1. В панели Beget прикрепить **дополнительный IPv4 из другой подсети** — облако
   поднимает его через cloud-init само, netplan руками не трогать.
2. В `/opt/mentors/.env.prod`: `SITE_ADDRESS=dev.<новый-ip-через-дефисы>.sslip.io`
   (sslip.io кодирует IP в имени — DNS менять не нужно), старое имя перенести в
   `SITE_ADDRESS_OLD` (блок 301-редиректа уже есть в `Caddyfile`), обновить
   `PLATFORM_URL` на новый адрес.
3. `$C up -d caddy web` — пересоздать оба с новым env; Caddy сам выпустит
   сертификат на новое имя.
4. Прогнать **чеклист доступности по операторам** (ниже) — сразу и через 24 ч.
5. После переходного периода: открепить старый IP в панели Beget и убрать
   блок `SITE_ADDRESS_OLD` из `Caddyfile`.

SSH-доступ не зависит от публичного IP (Tailscale-алиас `mentors-vps`).

### Переезд на постоянный домен

1. Купить домен; A-запись → текущий публичный IP (сейчас `155.212.211.251`),
   TTL на время переезда — 300.
2. `/opt/mentors/.env.prod`: `SITE_ADDRESS=<домен>`,
   `SITE_ADDRESS_OLD=dev.155-212-211-251.sslip.io` (301 со старого имени),
   `PLATFORM_URL=https://<домен>` (он же управляет ссылками в письмах и og-тегами).
3. `$C up -d caddy web` — Caddy выпустит сертификат Let's Encrypt на домен
   (порты 80/443 уже открыты; убедиться, что A-запись отвечает ДО рестарта,
   иначе ACME-челлендж не пройдёт — при неудаче просто повторить `up -d caddy`).
4. Проверить: `curl -sI https://<домен>/login | grep -i strict-transport`,
   og:url в исходнике страницы указывает на домен, вход работает.
5. Чеклист доступности по операторам (ниже) — сразу и через 24 ч.
6. Через 2–4 недели убрать `SITE_ADDRESS_OLD`-блок; sslip.io-имя умрёт само.

### Включение SMTP («когда решим»)

Сейчас email-канал **осознанно в лог-режиме** (решение старта, changelog 13.2):
`SMTP_HOST` пуст → nodemailer `jsonTransport`, письма пишутся в лог, ничего не
падает. Включение:

1. `/opt/mentors/.env.prod`: заполнить `SMTP_HOST`, `SMTP_PORT` (587),
   `SMTP_USER`, `SMTP_PASS`, `SMTP_FROM` (например `MENTORS <no-reply@домен>`).
2. `$C up -d web worker` — оба шлют почту (web — авторизационные письма,
   worker — outbox-рассылка уведомлений и дайджесты).
3. Проверка: на `/login` → «Забыл пароль» на свой адрес; письмо пришло — канал
   живой. Затем `$C logs worker | grep emailDispatch` — джоба тикает без ошибок.

До включения SMTP реквизиты доступа ученикам передаются лично (кнопка
«Копировать сообщение» в диалоге выдачи) — это штатный режим, не деградация.

### Выдача доступа ученику (от и до)

1. `/admin/students` → **«Выдать доступ»**: email (= логин), имя опционально
   (ученик задаст своё на онбординге).
2. Диалог результата показывает **временный пароль один раз** — «Копировать
   сообщение» и отправить ученику лично (Telegram/WhatsApp). Закрыл диалог —
   пароль больше нигде не всплывёт (хранится только хеш).
3. Первый вход ученика: активация (отсчёт **90 дней** идёт с первого входа,
   не с создания) → обязательная смена пароля → онбординг (имя → трек →
   время → напоминания).
4. Доступы разделов — в карточке ученика: тумблеры «Библиотека», «Резюме»,
   «Легенда» (у новых учеников по умолчанию **выключены** — включить по мере
   надобности).
5. Продление — карточка ученика («+30/+90») или массово в списке учеников
   (чекбоксы → «Продлить»). Заморозку стрика можно подарить там же.
6. Забыл пароль: сам — «Забыл пароль» на /login (нужен SMTP) — либо админ:
   карточка → «Сбросить пароль» → новый временный пароль (показывается один
   раз, тем же диалогом).
7. Подозрение на шаринг аккаунта: `/admin/security` (сессии, мультиустройства)
   → карточка → «Сбросить сессии» или «Заблокировать».

### Пятиминутка владельца (еженедельно)

Раз в неделю, Пульт (`/admin`) + два клика:

- **Пульт**: активные/WAU, последние активации, security-флаги (норма — 0),
  «Записи со ссылкой старше 30 дней: N» (N>0 → ротация ссылок, см. выше),
  аномалии воронки курсов.
- **/admin/security**: ученики с несколькими устройствами за 7 дней, странные
  сессии (город/устройство) — при сомнении «Завершить» сессию.
- **/admin/audit**: беглый взгляд на действия команды за неделю.
- **Сервер** (30 секунд): `ls -lht /opt/mentors/backups | head -3` — свежий
  бэкап сегодняшний; `df -h /` — на диске есть запас.

### Чеклист доступности по операторам (после ЛЮБОЙ смены адреса)

Обязателен после смены IP, домена или сертификата — урок ТСПУ-фильтрации:
сервер может быть жив, а часть операторов РФ его молча режет.

1. https://check-host.net → HTTP-проверка `https://<адрес>/login` — все
   российские ноды дают 200/301 (не таймаут).
2. **Два живых человека из разных сетей** (мобильный оператор + домашний
   провайдер, отличные от сети владельца) открывают `/login` и логинятся.
3. Частичная недоступность → зафиксировать операторов и время, снять трейс
   (`curl -v` у пострадавшего), дальше — процедура «Смена IP».
4. Повторить проверку **через 24 часа**: фильтры включаются с лагом.

### TLS 1.2-рычаг (заготовка — НЕ включать без диагноза)

Если у части операторов рвётся именно TLS-handshake (DPI, спотыкающийся о
TLS 1.3), в блок сайта в `Caddyfile` добавляется принудительное ограничение:

```caddyfile
{$SITE_ADDRESS} {
	tls {
		protocols tls1.2 tls1.2   # только TLS 1.2 — рукопожатие читаемо для DPI
	}
	encode zstd gzip
	reverse_proxy web:3000
}
```

Затем `$C up -d caddy`. Это **сужает** безопасность (отключает TLS 1.3) — рычаг
дёргать только при подтверждённой диагностике «TLS 1.3 не проходит у оператора
X», и откатывать, когда причина уходит. Сейчас НЕ включено: Caddy по умолчанию
отдаёт TLS 1.2+1.3, этого достаточно.
